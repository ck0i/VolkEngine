# Performance

## Current principles

- Keep hot paths explicit and measurable.
- Avoid per-frame heap allocation and repeated resource creation.
- Prefer persistent mapping for tiny per-frame CPU-to-GPU data.
- Use staging uploads for device-local geometry.
- Keep synchronization narrow and visible.
- Use Vulkan timestamp queries where supported, but allow `--no-gpu-timestamps` to remove their query pool and command writes from latency-focused benchmark runs.

## Already optimized

- Two frames in flight keep CPU frame preparation decoupled from GPU completion without unbounded latency.
- The no-vsync presentation path now prefers `VK_PRESENT_MODE_IMMEDIATE_KHR` before mailbox/FIFO to minimize present-queue latency on surfaces that expose immediate mode; vsync remains FIFO.
- Vulkan Memory Allocator owns buffer/image memory allocation, uses `VK_EXT_memory_budget` when available, and the ImGui overlay reports cached live resource counts plus VMA heap usage/budget totals refreshed on a short diagnostic interval instead of polling `vmaGetHeapBudgets` every frame.
- Scene uniforms are persistently mapped; frame updates compute the camera projection and view-projection matrices once, share that view-projection with a CPU-only visibility planning step, aggregate-initialize the six uniform fields directly from camera/light inputs, and perform one struct copy into mapped memory.
- Per-frame scene-instance data is persistently mapped in a descriptor-bound storage buffer that starts at a 2048-visible-instance floor and grows only after the owning frame fence has completed; growth is rejected if the visible compacted descriptor range would exceed `VkPhysicalDeviceLimits::maxStorageBufferRange`. Visibility planning culls once, counts visible items per mesh bucket, and prepares grid-cache work before command recording; command recording then computes prefix offsets and writes mesh-contiguous instance ranges directly into the mapped frame buffer so `firstInstance`/`instanceCount` remain correct without an O(meshes * visible-items) compaction loop or a second CPU scratch-to-GPU memcpy.
- Sandbox CLI validation caps generated demo scenes at 4,194,304 total items before creating a window; the renderer's per-frame device-limit validation now applies to visible compacted instances after culling, not total scene-list size.
- FrameGraph construction rejects exact duplicate `(pass, resource, access, usage)` edges up front, catching accidental duplicate dependency declarations without adding broad single-writer restrictions that would block legitimate multi-pass resource reuse.
- Per-frame primary command buffers now live in per-frame command pools; after the matching frame fence has completed and swapchain image acquisition succeeds, the renderer resets the whole frame pool with `vkResetCommandPool` and begins the command buffer with `VK_COMMAND_BUFFER_USAGE_ONE_TIME_SUBMIT_BIT`, matching Khronos command-buffer recycling guidance while keeping upload command pools separate.
- The frame fence is reset immediately before `vkQueueSubmit`, not before command recording. Pre-submit failures therefore leave the previously signaled fence intact and cannot poison the next reuse of the frame slot with an unsignaled fence that no GPU submission owns. If submit itself fails after the reset, the renderer preserves the original submit error, attempts a non-blocking replacement with a signaled fence, and throws; it does not wait on the just-reset fence.
- Present-wait semaphores are owned by swapchain images instead of frame slots. The frame still owns `imageAvailable`, but the graphics submit signals `swapchainRenderFinishedSemaphores_[imageIndex]` and present waits on that same image semaphore, avoiding reuse before the presentation engine has consumed it.
- Scene draw and triangle stats are derived from the reusable `SceneRenderList` and prepared visibility plan; stats count actual scene command submissions and visible triangles in the enabled scene passes plus the fullscreen tonemap draw, and expose the submitted prepass state, scene pass count, total/visible item counts, mesh-batch count, and indirect-vs-direct submission mode from the same plan materialized during command recording.
- Demo render items carry mesh IDs, material constants, transforms, and bounding spheres; visibility planning performs mesh lookup, culls visible items against planes extracted from the actual view-projection matrix, computes mesh-bucket counts, and leaves command recording to materialize that decision into instance/indirect buffers plus draw commands for the enabled scene pass or passes.
- Per-item culling uses a binary sphere-outside-frustum helper that exits on the first plane with `signedDistance < -radius`; the tri-state inside/intersects classifier remains reserved for grid-tile planning where fully-inside tiles and intersecting fallback paths need distinct handling.
- `SceneRenderItem` keeps `boundsCenter`, `boundsRadius`, and `mesh` at the front of the 144-byte record, with static offset checks, so visibility planning reads the fields it needs from the first cache line before command recording touches `Mat4` transform and material data for visible-item materialization.
- `DemoSceneRenderer` keeps unchanged material-grid and ground items resident in the returned `SceneRenderList`; repeated same-dimension frames patch only the five animated foreground items, preserving full-list `sceneItemCount`/renderer-culling semantics while removing millions of static per-frame item writes in large-grid stress runs.
- Material-grid metadata is an optional acceleration hint: visibility planning falls back to a full per-item scan for non-grid scenes or missing/stale tile coverage, while the demo grid precomputes 16x16 tile bounding spheres, homogeneous-mesh flags, and a cached full-coverage validity bit when the static layout changes. Fully outside tiles add their full item count to `culledDrawCalls`, homogeneous fully inside tiles add one compact work record instead of one visible-index record per item, mixed fully inside tiles fall back to per-item records, and intersecting tiles fall back to exact per-item sphere tests; `sceneItemCount = visible + culled` semantics stay unchanged.
- Homogeneous sphere tiles do not treat the tile's aggregate culling radius as one giant sphere for LOD. Each tile stores the maximum per-item bounds radius, and visibility planning combines that radius with conservative nearest-tile depth so fully accepted 16x16 tiles can still collapse to one work record without downgrading near-edge spheres or forcing far tiles to the high-poly sphere mesh.
- The default material-grid tile size remains 16x16 after post-LOD benchmarking: 1000x1000 smokes kept lower visibility-planning plus materialization time at 16x16 than 4x4 or 8x8, while the 8x8 triangle/GPU differences were too small to justify a universal heuristic. `--grid-tile-rows` and `--grid-tile-columns` expose tile granularity for scene-specific measurement without changing production defaults.
- Tile-cache generation uses remaining-row/column counts instead of `begin + tileSize` wrap-prone arithmetic. `SceneRenderList` clears the cached coverage bit on `clear()`/`setMaterialGridRange()`/failed rebuild and also invalidates cached tiles when mutable `operator[]` access targets an item inside the material-grid range; non-grid item edits keep the cache alive for the demo's animated foreground. After a valid rebuild has produced full tile coverage, visibility planning avoids a per-frame tile-coverage scan without trusting stale coverage state.
- Following the same temporal-coherence principle used by hierarchical visibility systems, the renderer caches only the static material-grid tile classification result for an unchanged view-projection matrix and monotonically increasing tile revision. The cache excludes animated foreground items, keeps mixed-tile fallback records exact during cache-miss planning, invalidates on camera/projection, range, tile-count, coverage, or revision changes, and stores packed static-grid `InstanceData` per mesh after command recording materializes the planned work so cache hits copy contiguous mesh chunks into the frame buffer instead of rematerializing every visible grid item. Cache misses retain only a scalar emitted-work count for telemetry, not a heap-backed copy of every compact grid work record. Repeated large static views therefore avoid retesting thousands of tile bounds and avoid rebuilding static grid instance payloads without changing visible-item, mesh-contiguous draw-range, or triangle-count semantics.
- `RenderStats`, the ImGui overlay, and the clean-exit log expose grid-tile culling telemetry (`gridTileCount`, accepted, culled, and intersected tiles) plus the material-grid visibility cache hit/miss bit, emitted grid work-record count, and GPU timestamp validity, making large-scene culling and timing availability visible without relying on a profiler capture or mistaking unavailable timestamp results for zero-cost GPU work.
- `--no-gpu-timestamps` is an intentional benchmark mode, not an error path: `createTimestampQueries()` leaves `timestampsEnabled_` false, so the existing write/readback branches skip query work and `gpuTimestampsValid` remains false.
- Mesh geometry is generated once, packed into one shared vertex buffer plus one shared index buffer, staged in host-visible buffers, and copied to device-local buffers through one batched transfer command buffer during initialization. Separate transfer queues signal a one-shot transfer-upload semaphore waited by the first graphics submit at `VERTEX_INPUT`; same-queue transfer/graphics devices skip that semaphore and use recorded `VkBufferMemoryBarrier2` transfer-write-to-vertex/index-read barriers instead, so both paths retain an explicit first-use dependency without same-queue semaphore churn.
- The scene path uses `vkCmdDrawIndexedIndirect` with a per-frame persistently mapped `VkDrawIndexedIndirectCommand` buffer when `multiDrawIndirect` and `drawIndirectFirstInstance` are enabled; otherwise the same shared geometry metadata feeds the direct `vkCmdDrawIndexed` fallback. This follows the Vulkan feature contract instead of making the existing legal direct `firstInstance` path depend on indirect-only feature bits.
- The loaded ground albedo texture is staged once during initialization and uploaded on the graphics queue into a device-local sampled image; mip generation uses in-place graphics-queue linear blits for supported formats, avoiding transfer/graphics queue-family ownership transfers for exclusive images.
- Graphics texture uploads and transfer-queue mesh uploads now share one upload submission/retirement helper; staging buffers, upload fences, and one-shot command buffers are retired from fence status. Mesh-upload wait semaphores are created only for separate transfer queues, collected into reused dynamic wait/stage vectors instead of a fixed-size array, and retained by the consuming frame until that frame fence completes; same-queue mesh uploads have no semaphore to collect or destroy.
- The depth prepass, HDR pass, and tonemap pass use dynamic rendering, avoiding framebuffer object churn; the default `--no-depth-prepass` path skips the depth-only rendering pass and uses a depth-writing HDR scene pipeline because the current demo is geometry-cost dominated, while `--depth-prepass` remains available for overdraw/material-cost experiments.
- Viewport and scissor dynamic state are emitted once after command-buffer begin and reused across the depth, HDR, and tonemap dynamic-rendering instances, avoiding redundant per-pass `vkCmdSetViewport`/`vkCmdSetScissor` calls while relying on Vulkan command-buffer-scoped dynamic state.
- Tonemapping exposure is a fragment push constant, avoiding descriptor churn for interactive HDR brightness tuning.
- Procedural environment lighting rides in the existing persistently mapped scene uniform buffer, so the IBL scaffold adds no descriptors, texture samples, or per-draw CPU work.
- `--no-imgui` removes the debug overlay from renderer initialization and frame-building boundaries, so latency runs can measure the HDR/tonemap path without Dear ImGui backend setup, per-frame `NewFrame`, diagnostics refresh, or overlay draw data.
- Screenshot readback remains off the normal frame path; when requested, the readback/copy path is labeled as the FrameGraph screenshot pass, and PPM output is staged to a same-directory temp file and published only after the file stream closes successfully, with unique-backup overwrite fallback to avoid truncated final screenshots or arbitrary `.bak` deletion on mid-publish failures.
- `vkDeviceWaitIdle` is not used in the normal frame loop.
- Pipeline cache data is persisted and header-validated before reuse, matching Vulkan guidance to avoid repeated pipeline compilation stalls across runs.
- Shader hot reload is opt-in and throttled; replacement pipelines are built before swapping, old pipeline sets are destroyed only after the tracked frame fences that could reference them signal, and failed reloads do not destroy the currently rendered pipeline set.
- Resize/swapchain recreation reuses existing graphics pipelines when attachment formats are unchanged; dynamic viewport/scissor state handles extent changes.
- Debug-utils object names are assigned during initialization, and per-frame pass labels use literal strings to avoid hot-path heap allocation.
- The throttled runtime window-title update uses a fixed stack buffer and `glfwSetWindowTitle` directly, avoiding `std::ostringstream`/temporary-string churn in the main loop.
- Dear ImGui uses its own backend-managed descriptor pool and is rendered after tonemapping in the existing swapchain dynamic-rendering block, avoiding HDR/depth pass changes and engine descriptor churn.
- Screenshot capture is opt-in and one-shot; the normal frame loop does not allocate readback buffers, wait on screenshot fences, or request swapchain transfer-src work unless `--screenshot FILE.ppm` is used. On screenshot frames, presentation is queued before the CPU waits for the copy fence and writes the PPM, avoiding an avoidable pre-present round trip while still finishing readback before swapchain teardown.
- Frame-graph pass/resource metadata is compiled once at startup with fixed arrays, then reused for pass labels/order assertions and access-aware image sync states without per-frame graph allocation. The graph is mode-specific for depth and includes the screenshot transfer-source readback edge as an optional path: default no-prepass runs compile three passes/five edges, while `--depth-prepass` runs compile four passes/six edges.
- Pipeline-cache saves write to a unique temporary sibling after successful cache readback/header validation, then publish with a last-writer-wins overwrite fallback and temp cleanup on failure; this avoids zero-length/partial cache files and prevents concurrent sandbox runs from clobbering the same `.tmp` path.
- The GPU resource registry uses fixed-size records and bounded copied names, so exposing live engine-resource counts plus total/buffer/owned-image/imported-image estimated bytes does not depend on dangling temporary strings or per-frame heap allocations.
- Scene submission avoids hot-path virtual dispatch: the demo scene produces data, and the Vulkan backend consumes fixed mesh-bucket arrays, cached per-batch triangle counts derived once from uploaded `GpuMesh` metadata, and compacted instance data in tight draw loops.
- Visibility planning uses one renderer-owned `visibleSceneWorkScratch_` vector that grows from actual visible work records, clears before command recording, and stores either individual visible items or compact homogeneous grid-tile work records. Instance data is then written directly into mesh-contiguous ranges of the mapped per-frame storage buffer, so steady-state frames reuse capacity without old visible-index/mesh-index/instance-data resize churn.
- The sandbox material grid is configurable with `--grid-rows/--grid-columns`; grids above the old 2048-item cap can now keep the 2048-instance GPU storage floor when culling leaves fewer visible instances, while still stress-testing CPU scene-list construction, culling, batching, and instance-buffer upload bounds.

## Worth optimizing next

- Add GPU occlusion/culling scaffolds after the now-shared geometry and indirect command path has enough scene-scale telemetry to justify the extra buffers, barriers, and readback/latency tradeoffs.
- Keep expanding resource accounting with per-resource residency detail, allocation statistics by resource class, and future defragmentation hooks for streaming resources.
- Extend the current 8/16-bit PPM/P6 texture loader to production image formats once texture streaming requirements are clearer.
- Evolve the static frame graph into the owner of barriers, transient resources, and pass dependencies.
- Build actual descriptor indexing/bindless descriptor-set layouts only after the resource model needs them; `RenderDeviceInfo` already exposes support-only capability bits for gating that path.
- Add draw submission benchmarks for larger mesh/material diversity so the renderer can decide when multi-draw indirect, command-buffer threading, or future GPU-driven submission pays for its synchronization cost.
- Add adaptive depth-prepass heuristics driven by measured overdraw/material cost on top of the current `DepthPrepassMode::ForceOff` / `DepthPrepassMode::ForceOn` plumbing; avoid per-frame toggling until there is a benchmark window and hysteresis.
- Extend the initialization upload path into a reusable streaming upload queue with ring-buffer staging, batched copy scheduling, and per-resource readiness tracking for runtime texture/mesh streaming.
- Add optional profiling markers around finer-grained renderer phases; RenderDoc-friendly object names already cover frame sync and one-shot upload command/fence/semaphore resources.

## Profiling notes

`RenderStats` exposes measured CPU render-submit milliseconds, wall-clock frame delta milliseconds, GPU frame milliseconds, GPU depth-prepass/HDR-scene/final-pass milliseconds with a `gpuTimestampsValid` flag for completed timestamp-query availability, draw count, triangle count, submitted prepass state, scene pass count, total/visible scene item counts, mesh-batch count, current scene-instance buffer capacity/MiB, sphere high/medium/low LOD instance counts, and elapsed time. The final-pass interval includes the fullscreen tonemap draw, optional ImGui overlay when compiled/enabled, and final swapchain transition to present because the timestamp is written after final swapchain rendering and the present-layout barrier. The timestamp path allocates one fixed four-query range per frame slot, stores the submitted prepass state on that frame slot, and masks deltas by the graphics queue's `timestampValidBits` before converting ticks to milliseconds. `waitIdle()` refreshes the most recently submitted frame slot before the clean-exit log, so the final pass breakdown corresponds to the last submitted frame when `gpuTimestampsValid` is true. The CPU scope starts after frame-fence wait/image acquisition and ends after graphics queue submission, so vsync, present pacing, acquire stalls, and resize waits remain visible in `frameDeltaMs` without being mislabeled as CPU work. Engine draw counts intentionally exclude ImGui overlay draw lists. The sandbox logs final stats on clean exit and updates the window title roughly twice per second with FPS, frame delta, CPU frame time, GPU frame time or `pending`, draw count, mesh-batch count, scene pass count, and culled count, avoiding per-frame title-string churn.

For latency runs that need less instrumentation overhead, `--no-gpu-timestamps` disables the timestamp path before pool creation; CPU timing, draw counts, culling counters, and clean-exit logs remain available.
The CPU timing fields are intentionally scoped to the same render-submit window as `cpuFrameMs`: scene build, frame preparation, `recordCommandBuffer`, and queue-submit setup/submission. The window starts after swapchain acquire and screenshot setup and ends after queue-submit bookkeeping; it excludes swapchain present/recreate handling and screenshot readback waits. Visibility planning and instance-capacity growth are part of the prepare bucket before command recording starts; the command-record bucket covers materializing the prepared visibility plan into mapped instance data, writing indirect commands, deriving stats, ImGui draw-data encoding, and Vulkan command encoding.

When `--no-depth-prepass` is active, depth-prepass time is reported as `0.000 ms` and HDR time spans the depth-writing color pass.

The GPU resource MiB values are renderer-owned/imported-resource estimates derived from image formats and buffer sizes, not authoritative GPU residency or total process memory. Startup logs, resize logs, and the overlay split the estimate into buffer bytes, owned image bytes, and imported swapchain image bytes so resize churn and render-target growth are visible. The estimate intentionally excludes ImGui backend internals, VMA allocator internals, and transient staging resources.

## External performance references

Current implementation cross-checks against external guidance:

- Khronos' wait-idle sample says `vkQueueWaitIdle`/`vkDeviceWaitIdle` are coarse synchronization that drains GPU work; normal rendering and successful shader hot reload use per-frame fences/semaphores, with `vkDeviceWaitIdle` limited to shutdown and resize/minimize teardown. Source: https://docs.vulkan.org/samples/latest/samples/performance/wait_idle/README.html
- Khronos' command-buffer usage sample recommends recycling command buffers by resetting command pools instead of frequent individual command-buffer resets, avoiding `VK_COMMAND_POOL_CREATE_RESET_COMMAND_BUFFER_BIT` for whole-pool recycling, and using `VK_COMMAND_BUFFER_USAGE_ONE_TIME_SUBMIT_BIT` for single-use recordings; frame command recording follows that pattern, while one-shot upload work uses separate transient upload pools. Same-queue mesh uploads use a `VkBufferMemoryBarrier2` transfer-write-to-vertex/index-read dependency instead of creating a semaphore that would only wait on the same `VkQueue`. Source: https://docs.vulkan.org/samples/latest/samples/performance/command_buffer_usage/README.html
- Khronos' multi-draw indirect sample recommends reducing CPU command-buffer overhead by placing many model draw commands in a GPU buffer, using one vertex/index binding for packed geometry, and executing the command array with multi-draw indirect when supported; the renderer now packs the generated meshes into shared geometry buffers and uses one `vkCmdDrawIndexedIndirect` per scene pass on devices that expose the required feature bits, while keeping a direct fallback for portability. Source: https://github.com/KhronosGroup/Vulkan-Samples/blob/main/samples/performance/multi_draw_indirect/README.adoc
- NVIDIA Vulkan guidance recommends parallel command-buffer recording, minimizing queue submissions, avoiding tiny command buffers, using `VK_COMMAND_BUFFER_USAGE_ONE_TIME_SUBMIT_BIT` for one-shot command buffers, and minimizing pipeline binds; the renderer already uses one primary frame command buffer and one queue submit per frame, keeps mesh state grouped, and should add secondary command-buffer threading or GPU-driven culling only when measured scene complexity justifies that synchronization and lifetime complexity. Source: https://developer.nvidia.com/blog/vulkan-dos-donts/
- The Vulkan `VkPresentModeKHR` reference defines `IMMEDIATE` as no-vblank/no-queue presentation that may tear, `MAILBOX` as vblanked single-entry replacement, and `FIFO` as the required queued vblank mode; the renderer maps `--no-vsync` to immediate-first/mailbox/FIFO fallback and keeps `--vsync` on FIFO. Source: https://docs.vulkan.org/refpages/latest/refpages/source/VkPresentModeKHR.html
- The Vulkan `VkPhysicalDeviceFeatures` reference states that `multiDrawIndirect` is required for indirect draw counts greater than one, and `drawIndirectFirstInstance` is required when indirect command structs carry nonzero `firstInstance`; the renderer enables and uses the indirect scene path only when both are supported, and `--no-indirect-draws` exercises the direct fallback. Source: https://docs.vulkan.org/refpages/latest/refpages/source/VkPhysicalDeviceFeatures.html
- The Vulkan `VkPhysicalDeviceFeatures` reference also gates anisotropic texture filtering behind `samplerAnisotropy`; the renderer queries that bit before logical-device creation, enables it only when supported, and caps the texture sampler's `maxAnisotropy` at `min(deviceLimit, 16x)` rather than assuming desktop-class filtering. Source: https://docs.vulkan.org/refpages/latest/refpages/source/VkPhysicalDeviceFeatures.html

- Vulkan Guide pipeline-cache guidance: https://docs.vulkan.org/guide/latest/pipeline_cache.html
- Vulkan Samples pipeline-management guidance: https://docs.vulkan.org/samples/latest/samples/performance/pipeline_cache/README.html
- Vulkan dynamic-rendering documentation: https://docs.vulkan.org/tutorial/latest/03_Drawing_a_triangle/02_Graphics_pipeline_basics/03_Render_passes.html
- Vulkan timestamp-query reference: https://docs.vulkan.org/refpages/latest/refpages/source/vkCmdWriteTimestamp2.html
- Vulkan Guide synchronization examples show `vkCmdPipelineBarrier2` with source/destination stages and access masks for producer-to-consumer buffer dependencies; the same-queue upload path follows that model with `TRANSFER_WRITE` as the producer and vertex/index input reads as the consumers. Source: https://docs.vulkan.org/guide/latest/synchronization_examples.html
