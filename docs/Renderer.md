# Renderer

## Vulkan backend overview

`VulkanRenderer` creates the instance, optional debug messenger, GLFW surface, physical device, logical device, queue handles, command pools, swapchain, depth image, HDR image, sampled scene texture, descriptors, pipelines, frame resources, meshes, timestamp queries, and Dear ImGui overlay backend. It targets Vulkan 1.3 and evaluates every physical device against concrete requirements before ranking: API 1.3, graphics/present/transfer queues, `VK_KHR_swapchain`, surface formats/present modes, and the `dynamicRendering`/`synchronization2` feature bits. If no adapter passes, startup throws one error listing each rejected adapter and its reasons. When `VK_EXT_debug_utils` is available, long-lived Vulkan objects are named and the depth prepass, HDR scene, and tonemap/ImGui command regions are labeled for RenderDoc and validation diagnostics.

The public renderer interface exposes `RenderDeviceInfo` as immutable metadata: backend type, adapter name, API version, max 2D image size, discrete/integrated classification, enabled capabilities for dynamic rendering, synchronization2, timestamps, validation, debug markers, memory budget, sampler anisotropy with the selected max anisotropy value, descriptor indexing, bindless sampled-image hardware support, multi-draw indirect support, indirect-first-instance support, Vulkan's `maxDrawIndirectCount`, whether indirect scene submission is enabled, and whether initialization selected same-queue upload barriers or a separate transfer-queue semaphore for staged mesh uploads. The descriptor-indexing fields are support-only metadata; the logical device does not enable bindless descriptor features until a real bindless resource model exists. This is intentionally small metadata, not a queue/allocator ownership split.
Startup logs mirror `RenderDeviceInfo` after timestamp-query creation, split into enabled capabilities, support-only limits, and the transfer-upload synchronization path so headless smoke output shows the actual backend path without implying bindless descriptors are enabled. The clean-exit log repeats the per-frame submission path and timing state needed to interpret the last rendered frame. `RenderStats::gpuTimestampsValid` separately marks whether per-frame GPU timing values came from completed timestamp queries; the window title, clean-exit log, and overlay show pending/unavailable instead of treating startup or unsupported timestamp results as zero-time GPU work.

`--no-gpu-timestamps` leaves timestamp query support disabled at `createTimestampQueries()`: no query pool is allocated, command recording skips timestamp reset/write commands through the existing `timestampsEnabled_` checks, readback returns `gpuTimestampsValid = false`, and startup logs `timestamps no`.

A fixed-capacity `FrameGraph` metadata plan is compiled once during renderer initialization against the configured depth-prepass mode. It always declares three image resources (depth, HDR color, imported swapchain) and a separate final present state for the swapchain. The default no-prepass graph compiles three passes and five edges: HDR scene writes depth plus HDR color, tonemap/ImGui samples HDR and writes the swapchain, and the optional screenshot-readback pass reads the swapchain as a transfer source. `--depth-prepass` compiles four passes and six edges: depth prepass writes depth, HDR scene reads that depth and writes HDR color, tonemap/ImGui writes the swapchain, and screenshot readback models the transfer-source copy used by `--screenshot`. Startup asserts that the depth edges and screenshot transfer-source edge are present, validates that non-imported resources are written before they are read, supplies pass names/colors for debug labels, and keeps present as an explicit final state instead of a fake pass.

FrameGraph edge declaration is intentionally strict at this stage: adding the same `(pass, resource, access, usage)` tuple twice throws immediately, while broader multi-pass resource reuse remains legal for future render-graph growth.

`GpuResourceRegistry` is a fixed-capacity CPU-side registry for long-lived engine-owned renderer allocations and imported swapchain images. It copies resource names into bounded storage, assigns stable IDs, unregisters through the buffer/image teardown paths, and reports live counts plus total/buffer/owned-image/imported-image estimated memory bytes to startup logs, resize logs, and the ImGui overlay. It does not claim total GPU memory usage: ImGui backend objects, VMA internals, and transient staging buffers are outside the live accounting.

## Frame loop

Each frame:

1. Wait for the current frame fence.
2. Read the previous fixed GPU timestamp range for that frame slot when timestamp queries are enabled (`frameStart`, `depthEnd`, `hdrEnd`, `finalEnd`).
3. Acquire the next swapchain image.
4. Update a persistently mapped uniform buffer with camera/light data.
5. Build a CPU-only visibility plan from the reusable `SceneRenderList`, then ensure the current frame's scene-instance storage and descriptor range can hold the visible compacted instance count before command recording begins.
6. Build the read-only ImGui stats overlay for this frame.
7. Reset the current frame's command pool after the frame fence has completed, then record a one-time-submit primary command buffer with synchronization2 image barriers and scoped debug-utils labels for each render pass.
8. Submit with the current frame's image-available semaphore and the acquired swapchain image's render-finished semaphore.
9. Present.

Normal rendering does not call `vkDeviceWaitIdle`.

`imageAvailable` semaphores are frame-slot resources because they are waited by the next graphics submit for that CPU frame. Present-wait `renderFinished` semaphores are swapchain-image resources, selected by the acquired `imageIndex` and recreated with the swapchain, because only reacquiring the same image proves the previous `vkQueuePresentKHR` wait on that image has completed.

The frame fence is deliberately reset at the submit boundary, after uniform updates, command-pool reset, ImGui encoding, command recording, and wait-semaphore collection have succeeded. The renderer snapshots the depth image, HDR image, and acquired swapchain image `ImageSyncState` values before command recording, then restores that snapshot on any exception before a successful `vkQueueSubmit`; the tracked layouts/stages/access masks are only committed once the recorded transitions have actually been queued. If `vkQueueSubmit` itself fails after the fence reset, the renderer restores those image sync states, attempts to create a replacement signaled fence, swaps it onto the frame, and throws the original submit error. Screenshot write/readback and present failures occur after a successful submit, so they leave the submitted image sync state committed.

One-shot mesh uploads keep synchronization explicit without carrying dead wait objects into normal frames. When the selected transfer queue differs from the graphics queue, upload submits signal one-shot semaphores and the next graphics submit waits on the pending set at `VK_PIPELINE_STAGE_VERTEX_INPUT_BIT`; the wait list uses renderer-owned vectors with retained capacity instead of a fixed compile-time semaphore cap. When both roles resolve to the same `VkQueue`, the upload command buffer records `VkBufferMemoryBarrier2` barriers from `TRANSFER_WRITE` to `VERTEX_ATTRIBUTE_READ` and `INDEX_READ`; queue order plus those barriers satisfy visibility without a semaphore.

## Swapchain handling

The swapchain prefers BGRA/RGBA UNORM with `VK_COLOR_SPACE_SRGB_NONLINEAR_KHR` when available because the tonemap shader manually gamma-encodes LDR output; choosing an sRGB image format as well would double-encode. Swapchain suitability requires color-attachment image usage, optional transfer-dst usage is requested only when the surface advertises it, and optional transfer-src usage is requested for one-shot `--screenshot FILE.ppm` readback when available. `--no-vsync` prefers `VK_PRESENT_MODE_IMMEDIATE_KHR` for minimum present-queue latency, falls back to mailbox when immediate is unavailable, and finally falls back to FIFO; vsync uses FIFO. Resize/minimize handling waits until the framebuffer has non-zero dimensions, idles the device for safe teardown, recreates swapchain image views and HDR/depth targets, updates the tonemap descriptor to the new HDR view, refreshes the ImGui Vulkan backend so dynamic-rendering color format and image counts match the new swapchain, and reuses existing scene/tonemap pipelines when the HDR, depth, and swapchain formats are unchanged.
After a successful or suboptimal present, the renderer resets that swapchain image's tracked `ImageSyncState` to undefined. The next acquire therefore transitions from `VK_IMAGE_LAYOUT_UNDEFINED` with a don't-care load instead of assuming the presentation engine preserved `PRESENT_SRC_KHR`, which keeps the tracked-barrier model valid across drivers that discard or retile presented images.

The zero-sized framebuffer wait uses GLFW event blocking instead of a busy poll and aborts the recreate when a close event arrives while minimized, so a minimized close does not spin a CPU core or wait forever for a resize.

## Resource management

Buffers and images are explicit structs containing Vulkan handles plus VMA allocation handles. VMA performs buffer/image memory allocation, tags allocations for diagnostics, and enables `VK_EXT_memory_budget` when the selected device exposes it. Image transitions use a tracked `ImageSyncState` (`layout`, stage, access) so same-layout hazards such as depth write-to-read cannot drift away from the renderer's cached state. Mipmap generation intentionally keeps raw per-subresource barriers while different mip levels occupy different layouts, then marks the whole texture sampled-read once generation completes. Meshes are uploaded during initialization through host-visible VMA staging buffers submitted to the selected transfer queue; all demo mesh vertex/index copies are recorded into one one-time transfer command buffer and waited with one fence before staging buffers are destroyed. One-time graphics/transfer upload helpers destroy their fences and free command buffers on both success and checked-failure paths. Device-local mesh buffers use concurrent graphics/transfer sharing only when those queue families differ. Per-frame uniform buffers and instance storage buffers are host-visible/coherent and persistently mapped for the lifetime of the frames in flight.

Transient staging/readback maps use a small RAII guard so exceptions cannot leave VMA allocations mapped. Startup texture and mesh staging keep those guards in tight scopes around the `memcpy`; the maps are released before staging buffers are moved into upload-batch ownership and before submit-path cleanup is allowed to destroy them.
Renderer startup is exception-safe across partially created Vulkan and ImGui state. The constructor tears down any resources created before a failure without persisting a possibly stale pipeline cache, while normal shutdown persists the cache before destroying it. Dear ImGui initialization has its own local rollback path because its Vulkan backend can throw through `CheckVkResultFn` before the renderer marks the overlay as fully initialized.

Each frame owns its own graphics command pool and primary command buffer. After the frame fence signals and image acquisition succeeds, the renderer resets that frame's pool with `vkResetCommandPool` and re-records the primary command buffer with `VK_COMMAND_BUFFER_USAGE_ONE_TIME_SUBMIT_BIT`. One-shot graphics/transfer uploads keep using separate upload command pools, so per-frame pool resets cannot invalidate staging or mip-generation command buffers.

Initialization uploads share one submission/retirement path for graphics texture uploads and transfer-queue mesh uploads. Texture uploads submit on the graphics queue and rely on queue order plus recorded image barriers before first sampling. Mesh uploads submit on the selected transfer queue. If that queue is separate from graphics, each upload submit signals a one-shot semaphore and the first graphics frame waits on every pending upload semaphore at `VK_PIPELINE_STAGE_VERTEX_INPUT_BIT` before vertex/index fetch; the wait and stage arrays are scratch vectors reused across frames so upload bursts are bounded by memory, not by `kMaxFramesInFlight` or a small fixed cap. If transfer and graphics are the same queue family, no semaphore is created; the transfer command buffer records buffer barriers before the graphics command buffer consumes the data.


## Pipeline and shader system

CMake compiles every `engine/shaders/*.vert` and `*.frag` file into `${binaryDir}/shaders/*.spv` using `glslc --target-env=vulkan1.3 -I engine/shaders -MD -MF <depfile> -MT <spv>`; shared GLSL files under `engine/shaders/common/*.glsl` hold the scene uniform ABI, instance-buffer ABI, and PBR helper code, and compiler-generated depfiles track the exact includes each root shader consumes instead of making every shader depend on every common file. The `VolkEngineRuntimeShaders` target copies those SPIR-V files next to the sandbox executable on every build with explicit dependencies on the shader outputs. Runtime shader loading reads those SPIR-V files, rejects bytecode that is too small for a SPIR-V header, not a 32-bit word multiple, byte-swapped, or missing the `0x07230203` magic number, then copies the byte payload into an aligned `std::uint32_t` word vector before calling `vkCreateShaderModule`. A Vulkan pipeline cache object is shared by graphics pipelines, saved under `${binaryDir}/cache/pipeline_cache.bin`, and reloaded on the next run only when the cache header matches the selected vendor, device, and pipeline-cache UUID. Saves read back a complete cache payload, validate the generated header, write `pipeline_cache.bin.tmp`, and publish it only after the temporary file is fully written so a failed write does not truncate the previous cache.

When `--hot-reload-shaders` is enabled, the renderer polls those copied SPIR-V files at a throttled interval. A change builds a complete replacement pipeline set into local handles first; only after every replacement pipeline/layout is created successfully does the renderer swap the new handles into use, queue the previous set with the fences for frame slots that could still reference it, destroy that old set only after those fences signal, and refresh the stored file timestamps. Failed rebuilds log a warning and keep the existing pipelines live.

`SceneUniforms` and `InstanceData` have compile-time size/offset assertions matching the GLSL uniform block and storage-buffer instance layout. This catches C++/shader ABI drift at compile time instead of as corrupted GPU data.

Runtime assets are copied to `${binaryDir}/assets` and exposed through `EngineConfig::assetDirectory`. The current texture path loads `assets/textures/ground_albedo.ppm` with a small binary PPM/P6 decoder, scales 8-bit or 16-bit RGB samples into RGBA8 with malformed-sample checks, creates a `TRANSFER_SRC|TRANSFER_DST|SAMPLED` sRGB image, uploads mip 0 through a graphics-queue staging command buffer, generates the remaining mip levels with checked linear blits when supported, falls back to one mip otherwise, creates the repeat texture sampler after the texture so `maxLod` clamps to the actual mip chain, enables `samplerAnisotropy` only when the physical device exposes it, and binds the resulting image at scene set binding 1 for fragment sampling.

The tonemap pipeline uses a small fragment push constant for exposure, configurable through the sandbox `--exposure` flag, before ACES approximation and gamma encoding.

Dear ImGui v1.92.8 is built as a separate support library from source. Its Vulkan backend uses `UseDynamicRendering = true` with `PipelineInfoMain.PipelineRenderingCreateInfo` set to the current swapchain color format, a backend-owned descriptor pool, and the renderer's pipeline cache. The overlay is drawn after the fullscreen tonemap triangle while the swapchain image is still in `VK_IMAGE_LAYOUT_COLOR_ATTACHMENT_OPTIMAL`, then the image transitions to `PRESENT_SRC_KHR`. It shows frame/CPU/GPU timings with timestamp-validity handling, scene/LOD/grid stats, swapchain mode, validation state, GPU class, transfer-upload sync path, indirect-draw limit, descriptor-indexing/bindless support, sampler-anisotropy state, tracked resource bytes, and cached VMA memory-budget data. Slowly changing diagnostics such as resource registry counts and VMA heap budgets are cached and refreshed on a short interval so the overlay does not call the registry and `vmaGetHeapBudgets` every frame.

Starting the sandbox with `--no-imgui` leaves `EngineConfig::debugOverlay` false, skips Dear ImGui creation, compiles the final graph pass as tonemap-only, and makes both `beginImGuiFrame()` and `renderImGui()` no-op before touching ImGui backend state. This keeps latency benchmarks from paying overlay context, diagnostics, `NewFrame`, or draw-data costs.

Screenshot capture is an explicit diagnostics path, not a normal frame-loop dependency: `VulkanRenderer::requestScreenshot` mutex-protects the pending path, `draw()` consumes one request into a local path before doing Vulkan work, allocates a transient host-visible/coherent transfer-dst readback buffer with VMA random host access, records a copy from the final LDR swapchain image before the existing present-layout transition, queues presentation on the render-finished semaphore before the CPU waits for the frame fence, then maps the readback allocation behind a scope guard so file-open/write exceptions cannot leave it mapped, writes tightly packed BGRA/RGBA8 pixels as binary PPM/P6 RGB, and destroys the readback buffer before any swapchain recreation triggered by that present result.
The screenshot readback/copy path is wrapped in the explicit FrameGraph screenshot pass debug label; the final present-layout transition stays outside that label so debug captures do not attribute normal presentation work to screenshot readback.

Screenshot PPM output is staged through a same-directory `.tmp` file and only published after the stream is closed and checked. This is complete-before-publish behavior rather than a platform-specific atomic replacement API; when a platform cannot rename over an existing screenshot, the existing file is first moved to a unique sibling backup, restored if publishing the new temp file fails, and deleted only after the new file is safely in place. If restore itself fails, the backup and temp paths are kept for recovery.

## Scene path

The demo scene currently renders 26 fixed render items:

- a rotating rough metal sphere;
- a rough dielectric cube;
- three smaller material-calibration spheres spanning rough dielectric, mixed, and polished metallic responses;
- a 4x5 sphere grid that varies roughness by row and metallic response by column to exercise real instanced batching;
- a ground plane with a loaded albedo texture plus shader-generated grid tint.

Implementation note: instance compaction is bucketed by `SceneMeshId`, not by scene order. Visibility planning counts visible items per mesh; command recording computes prefix offsets from those counts and fills per-mesh contiguous ranges with write cursors so each scene batch receives a contiguous `firstInstance`/`instanceCount` range.

The three generated demo meshes now live in one renderer-owned scene vertex buffer and one scene index buffer. `GpuMesh` is metadata only (`indexCount`, `firstIndex`, `vertexOffset`), which avoids double ownership and lets both paths bind geometry once per scene pass. When the logical device enables both Vulkan core features required by the spec for this use (`multiDrawIndirect` for `drawCount > 1`, and `drawIndirectFirstInstance` for nonzero indirect `firstInstance` fields), command recording writes one persistently mapped per-frame `VkDrawIndexedIndirectCommand` array and submits visible mesh batches with one `vkCmdDrawIndexedIndirect` per scene pass. If either feature is absent, or the sandbox is launched with `--no-indirect-draws`, the renderer keeps the legal direct fallback and emits one `vkCmdDrawIndexed` per visible mesh batch using the same shared buffer offsets.
When indirect scene draws are requested but cannot be enabled, startup emits a warning naming the missing feature or `maxDrawIndirectCount` limit before falling back to direct indexed-instanced draws.

The shader path is a compact Cook-Torrance GGX forward pass. `DemoSceneRenderer` builds a CPU-only
`SceneRenderList` containing mesh IDs, model matrices, material constants, and bounding spheres. Before
command recording, `VulkanRenderer` resolves those mesh IDs to mesh-bucket IDs, extracts frustum planes from
the actual camera view-projection matrix, culls render-item bounding spheres once, computes visible mesh
counts, and prepares grid-cache work records. It then grows the current frame's persistently mapped
descriptor-bound instance buffer only when the visible compacted count exceeds that frame's capacity. Command
recording materializes the visibility plan into mesh-contiguous instance ranges, writes per-frame indirect
commands when supported, and submits either one indirect scene draw or direct indexed-instanced fallback draws
to each enabled scene pass. The per-frame instance buffer starts at a 2048-visible-instance floor, rejects
visible storage-buffer ranges above `VkPhysicalDeviceLimits::maxStorageBufferRange`, and reuses that storage
across frames. With `--depth-prepass` the depth-only pass writes depth before the HDR color pass loads it;
with `--no-depth-prepass`, the HDR scene pass writes depth directly through a separate depth-writing scene
pipeline. The vertex shader indexes the compacted instance array with Vulkan `firstInstance`, so batched
draws do not read stale per-draw state. Lighting combines one directional source with a procedural sky/ground
environment scaffold stored in `SceneUniforms`; the shader applies it as indirect diffuse plus
roughness-weighted ambient specular.

Instance materialization now writes directly into the current frame's persistently mapped storage buffer after mesh prefix offsets are known; the remaining CPU scratch is only compact visibility work records, not a second `InstanceData` staging array copied into mapped GPU-visible memory.

For unchanged material-grid dimensions, `DemoSceneRenderer` keeps the static grid and ground resident inside the returned list and only rewrites the five animated foreground items. This preserves full-list renderer culling/stat semantics while making large-grid multi-frame runs measure visibility planning, cache reuse, and command materialization instead of repeated scene generation.
The material-grid range stored on `SceneRenderList` is optional metadata for this demo path. When present with cached tile coverage, visibility planning trusts the list's prevalidated coverage flag and tests precomputed 16x16 grid-tile bounding spheres first: outside tiles are skipped as culled, inside tiles are accepted without per-item frustum tests, and intersecting tiles fall back to exact per-item sphere tests. Tile metadata also records whether the tile is homogeneous by `SceneMeshId`; homogeneous accepted tiles are represented as one compact work item and expanded only during instance materialization, while mixed tiles keep the per-item fallback. Missing, invalid, or out-of-range tile coverage leaves the flag false and falls back to the generic full-list scan instead of skipping the grid.

`DemoSceneRenderer` defaults to 16x16 material-grid tiles because current 1000x1000 LOD/culling smokes keep the lowest visibility-planning plus materialization cost there; `--grid-tile-rows` and `--grid-tile-columns` intentionally expose tile dimensions as measurement controls rather than an adaptive heuristic. Changing tile dimensions is part of the demo scene layout cache key, so same-sized grids rebuild tile metadata when the benchmark changes granularity.

Mutable access through `SceneRenderList::operator[]` preserves the cached tile metadata for items outside the material-grid range, but clears cached tiles and coverage when a grid item is touched; the next `rebuildMaterialGridTiles()` restores the fast path from the mutated data.

`VulkanRenderer` also keeps a temporal visibility cache for the static material-grid tile classifications. The cache is keyed by the exact camera view-projection matrix, material-grid range, tile count, and `SceneRenderList::materialGridTileRevision()`; foreground animated items and trailing non-grid items are still planned every frame. Cache misses emit the compact grid work records needed for the current frame, store only a scalar emitted-work count for telemetry, and cache the already packed static-grid `InstanceData` per mesh after command recording materializes the plan. Cache hits then materialize foreground work, copy each mesh's cached grid instances into the current frame's mesh-contiguous range, and materialize trailing non-grid work without inserting cached grid records back into the per-frame visible-work scratch vector. Any camera/projection change, grid-range change, tile-count change, stale coverage flag, or tile revision change recomputes the grid path instead of reusing stale visibility or instance data.

For homogeneous sphere tiles, the compact grid work record targets a renderer sphere LOD batch rather than the logical `SceneMeshId` directly. Item-level sphere draws score LOD from their own bounds radius and view depth; homogeneous tiles score from `SceneGridTile::maxItemBoundsRadius` and a conservative nearest-tile depth (`tileCenterDepth - tile.boundsRadius`) so near items inside a 16x16 tile are not downgraded by the tile center. `tile.boundsRadius` remains the aggregate culling sphere and is not used as the projected sphere size.

`RenderStats` also reports grid-tile culling partition counts, material-grid visibility cache state, current frame instance-storage capacity/MiB, and sphere high/medium/low LOD instance counts. The clean-exit line preserves the stable `visible {}/{}, draws {}, culled {}, triangles {}` sequence and appends `grid tiles N (accepted A, culled C, intersected I), grid cache hit|miss (work W), instance cap N (M MiB), sphere LOD instances H/M/L` for headless smoke logs.

Runtime stats report two engine draw commands by default on hardware with indirect scene draws enabled when all 26 scene objects are visible: one multi-draw indirect HDR scene command plus the fullscreen tonemap triangle. With `--depth-prepass`, the same default scene reports three engine draw commands because the visible mesh batches are submitted once to the depth prepass and once to the HDR scene pass. `--no-indirect-draws` reports the portable direct fallback counts: four commands without a depth prepass and seven with a depth prepass. `RenderStats` also records the actual submitted prepass state, scene pass count, scene item count, visible item count, mesh-batch count, and whether indirect scene submission was used from the prepared visibility plan and recorded mesh-batch list; per-frame resources store the submitted prepass mode used for timestamp splitting, so later adaptive selection can vary by frame without corrupting timing. ImGui overlay draw lists are intentionally excluded from `RenderStats::drawCalls` so scene renderer instrumentation remains stable.
The sandbox clean-exit log repeats the submission mode as `submission multi-draw-indirect` or `submission direct`, so CI logs and RenderDoc repro notes carry the same path label as the ImGui overlay without requiring a screenshot.

CPU timing is sampled inside the renderer's existing render-submit window: it starts after swapchain acquire and screenshot-request setup, and it ends immediately after `vkQueueSubmit` plus upload-semaphore bookkeeping. `RenderStats::cpuFrameMs` is the total of that window; `cpuSceneBuildMs`, `cpuPrepareMs`, `cpuCommandRecordMs`, and `cpuQueueSubmitMs` split it into mutually exclusive buckets. The prepare bucket covers CPU visibility planning, mesh lookup, frustum/tile culling, visible-count capacity growth, and ImGui stats refresh. The `record` bucket is `recordCommandBuffer`, so it materializes the prepared work into mapped instance/indirect buffers, derives final command stats, encodes ImGui draw data, and emits Vulkan commands; it is not a pure Vulkan command-buffer API cost. Present/recreate handling and screenshot readback waits remain outside the total.
